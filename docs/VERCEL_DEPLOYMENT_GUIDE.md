# Vercel Deployment Guide

## 1. Create the Vercel project

1. Import the GitHub repository into Vercel.
2. Select the Next.js framework preset.
3. Keep the root directory as the repository root.
4. Use the default install command: `npm install`.
5. Use the production build command: `npm run build`.
6. Use the output preset generated by Next.js.

## 2. Configure environment variables

Set the following for Production and Preview:

- `NEXT_PUBLIC_SITE_URL`
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

Set mobile variables only when building the Expo app pipeline:

- `EXPO_PUBLIC_API_BASE_URL`
- `EXPO_PUBLIC_SUPABASE_URL`

Production `NEXT_PUBLIC_SITE_URL` must match the canonical public domain, for example:

```bash
NEXT_PUBLIC_SITE_URL=https://uvelyglow.com
```

## 3. Configure domains

1. Add the production domain in Vercel.
2. Configure DNS records as instructed by Vercel.
3. Set the same domain in Supabase Auth redirect URLs.
4. Update `NEXT_PUBLIC_SITE_URL` to the final HTTPS URL.
5. Redeploy after domain changes.

## 4. Configure build settings

Required checks:

```bash
npm run typecheck
npm run lint
npm run build
```

The repository currently builds as a standard Next.js 15 app. The Expo scaffold under `apps/mobile` is documentation/architecture only and is not part of the Vercel web build.

## 5. Image optimization

`next.config.ts` allows:

- `images.unsplash.com`
- Supabase Storage host derived from `NEXT_PUBLIC_SUPABASE_URL`

Before launch, ensure every production product, blog, brand, review, and media image is hosted on one of these allowed hosts or update `next.config.ts`.

## 6. Production deployment flow

1. Merge launch-ready changes to `main`.
2. Confirm Supabase production migrations have been applied.
3. Confirm Vercel environment variables are set.
4. Trigger Vercel production deployment from `main`.
5. Watch build logs until completion.
6. Run post-deployment smoke tests from `docs/DEPLOYMENT_CHECKLIST.md`.
7. Check Vercel runtime logs for errors after smoke tests.

## 7. Rollback plan

- Use Vercel's previous deployment rollback for frontend regressions.
- For database issues, prefer forward-fix migrations.
- Do not roll back Supabase migrations by deleting tables in production unless a backup restore plan has been approved.
- Keep a copy of the last known good deployment URL.

## 8. Recommended Vercel hardening

- Require preview deployment protection for non-public branches.
- Enable deployment comments/checks in GitHub.
- Add uptime monitoring for the production domain.
- Add a log drain before public launch.
- Separate Preview and Production Supabase projects to avoid test data contamination.
